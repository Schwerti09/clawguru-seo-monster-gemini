## 4. DAS HARDCODING-PROBLEM (kritisch)

### Ausmass
- **283 Dateien** unter `app/` mit `locale === "de"` Checks
- **254 Dateien** unter `app/[lang]/` mit `isDE ?` Checks
- **17 Komponenten** mit hardcoded `isDE ?` Checks
- **Verhältnis:** Dictionary wird in ~25 Dateien genutzt, Hardcoding in ~300 Dateien

### Was das bedeutet
Wenn jemand die Seite auf Französisch besucht und eine Seite nur `isDE ? "..." : "..."` hat,
bekommt er Englisch — aber NICHT Französisch.
Das Dictionary-System hilft ihm gar nicht, weil es gar nicht aufgerufen wird.

### Häufigste Hardcoding-Muster
```tsx
// Muster 1 — Ternary (häufigste Form)
{isDE ? "Sicherheits-Check starten" : "Start Security Check"}

// Muster 2 — locale check direkt
const title = locale === "de" ? "Titel DE" : "Title EN"

// Muster 3 — in Metadata
title: locale === "de" ? "ClawGuru | Sicherheit" : "ClawGuru | Security"
```

## 8. AUTOMATISIERUNGS-STRATEGIE

Da 1011 × 16 = 16.176 Key-Übersetzungen zu verwalten sind, brauchen wir ein automatisiertes System:

### Geplantes Skript: `scripts/translate-missing.js`
```
1. Lade EN.json als Referenz
2. Für jede Sprache: finde fehlende Keys
3. Sende fehlende Keys in Batches an Gemini API
4. Schreibe Ergebnisse zurück in {locale}.json
5. Log: welche Keys wurden übersetzt, welche übersprungen
```

### Quality-Assurance
- Back-Translation Spot-Check (10% der Keys)
- Native-Speaker Review für DE (Schwerti kann prüfen)
- Auto-Flag bei Keys, die länger als 150% des EN-Keys sind (möglicherweise falsche Übersetzung)
