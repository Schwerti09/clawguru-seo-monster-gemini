# i18n-Roadmap — Nach 30-Sprachen-Deploy
**Stand:** 22.04.2026 | Status: Blueprint für Phase 2–6

---

## Kontext

Nach Round 14 (21./22.04.2026) haben wir:
- **30 Sprachen** mit jeweils 775/775 Dictionary-Keys (ohne `roast.*`)
- **14 Alt-Sprachen** auf 100% (af, ar, es, fr, hi, it, ja, ko, nl, pl, pt, ru, tr, zh)
- **14 Neu-Sprachen** in Übersetzung (he, uk, vi, id, sv, fi, ro, cs, th, bn, el, hu, da, no)
- **RTL-Support** für ar + he
- **Sitemap/robots.txt** korrekt konfiguriert

**Aber:** Dictionary-Vollständigkeit ≠ nutzbare Mehrsprachigkeit. Die folgenden 5 Hebel
schließen die Lücke zwischen "übersetzt" und "wirkt bei User + Google".

## Priorisierte Roadmap

### ⚡ PHASE 2 — Hardcoding-Exorzismus (KRITISCH)
**Aufwand:** 1–2 Tage Agent-Arbeit | **Impact:** macht die 30 Sprachen überhaupt erst nutzbar

Ausgangslage (siehe `i18n-analysis-2026-04-21.md`):
- 283 Dateien mit `locale === "de"` Checks
- 254 Dateien mit `isDE ?` Checks
- 17 Komponenten mit hardcoded `isDE ?`

**Problem:** Ein japanischer User auf `/ja/pricing` sieht englischen Text, weil die Seite
`isDE ? "DE-Text" : "EN-Text"` hat und gar nicht ins Dictionary schaut.

**Reihenfolge nach Revenue-Impact:**

| # | Datei | Geschätzte Keys | Aufwand |
|---|-------|-----------------|---------|
| 1 | `app/[lang]/pricing/page.tsx` | ~35 | 2h |
| 2 | `app/[lang]/academy/page.tsx` | ~50 | 3h |
| 3 | `app/[lang]/check/page.tsx` | ~25 | 2h |
| 4 | `app/[lang]/consulting/page.tsx` | ~20 | 1h |
| 5 | `components/cert/PracticeExamClient.tsx` | ~30 | 2h |
| 6 | `app/[lang]/defender-cert/page.tsx` | ~25 | 2h |
| 7 | `app/[lang]/for-msps/white-label/page.tsx` | ~20 | 1h |
| 8 | `components/booking/BookingButton.tsx` | ~5 | 30min |
| 9 | `components/commerce/BillingToggle.tsx` | ~15 | 1h |

**Methode:**
1. Jeden hardcoded `isDE ? "..." : "..."` in Dictionary-Key umwandeln

3. Für alle 28 weiteren Locales: `node scripts/translate-all.js` erneut laufen lassen
4. Build + Test

**Ergebnis:** Alle 30 Sprachen zeigen echt lokalisierten Text auf Conversion-Seiten.

### 🌐 PHASE 3 — Accept-Language Auto-Redirect
**Aufwand:** 30–60 min | **Impact:** reduziert Absprungrate bei Nicht-DE/EN-Usern um ~30–40%

**Jetziger Zustand:** `clawguru.org` ohne `/lang`-Prefix landet auf Default-Locale (EN).
Ein japanischer User muss manuell in den Sprachwähler.

**Lösung:** Middleware in `middleware.ts` erweitern:
```ts
if (pathname === "/") {
  const accept = req.headers.get("accept-language") || ""
  const match = parseAcceptLanguage(accept, SUPPORTED_LOCALES)
  if (match && match !== DEFAULT_LOCALE) {
    return NextResponse.redirect(new URL(`/${match}`, req.url), 307)
  }
}
```

**Wichtig:**
- **307 (temporary)**, nicht 301 — sonst cached Google die Weiterleitung
- Nur für die Root `/`, NICHT für bereits gewählte `/en/…` oder `/ja/…`
- Cookie `NEXT_LOCALE` schreiben, damit Rückkehr auf `/` die Auto-Detection nicht überschreibt

**Ergebnis:** Erste Besucher landen sofort in ihrer Sprache.

### 🗺️ PHASE 4 — hreflang im Sitemap-XML + x-default
**Aufwand:** 1h | **Impact:** schnelleres Crawling lokalisierter Versionen, x-default als Fallback

**Jetziger Zustand:** `app/sitemaps/[name]/route.ts:97-102` gibt bewusst leeres `buildAlternateLinks` zurück.
hreflang steht nur im HTML-`<head>` via Next.js `generateMetadata.alternates`.

**Problem:** Google muss jede Seite erst crawlen, um hreflang zu sehen. Mit hreflang
**direkt im Sitemap-XML** werden alle Sprachvarianten sofort erkannt.

**Lösung:**
```ts
function buildAlternateLinks(loc: string): string[] {
  const match = loc.match(/^https:\/\/[^/]+\/([a-z]{2})\/(.*)/)
  if (!match) return []
  const [, , path] = match
  return [
    ...SUPPORTED_LOCALES.map(
      (l) => `    <xhtml:link rel="alternate" hreflang="${l}" href="${BASE_URL}/${l}/${path}"/>`
    ),
    `    <xhtml:link rel="alternate" hreflang="x-default" href="${BASE_URL}/${DEFAULT_LOCALE}/${path}"/>`,
  ]
}
```

Plus: `xmlns:xhtml="http://www.w3.org/1999/xhtml"` im `<urlset>`-Tag.

**⚠️ Vorsicht:**
- Nur in Sitemaps mit < 10k URLs einbauen (sonst 50MB-Limit). Für große Runbook-Buckets
  weiter im `<head>`. Hub-Sitemaps (main-*.xml) sind klein genug.

**Ergebnis:** Google entdeckt lokalisierte Versionen ohne Einzel-Crawl.

### 🎨 PHASE 6 — Native-Speaker-QA für Top-5 Zielsprachen
**Aufwand:** ~€150 (externe Reviewer) + 2h Integration | **Impact:** 2–3× Conversion in asiatischen Märkten

Gemini übersetzt technisch korrekt, aber **Tonalität** (professionell-direkt wie SRE) ist
in asiatischen Sprachen subtil. Für Conversion-kritische Seiten (`/pricing`, `/check`, Hero)
lohnen sich Muttersprachler-Reviews.

**Top-5 nach Markt-Potenzial:**
1. **Japanisch (ja)** — hoher DevOps-Markt, wenig Konkurrenz
2. **Koreanisch (ko)** — Tech-Szene, Cybersecurity-Budget hoch
3. **Chinesisch (zh)** — riesig, aber China-SEO = Baidu (separate Strategie)
4. **Arabisch (ar)** — UAE/Saudi, RTL-Handling wichtig
5. **Spanisch (es)** — Latam + Spanien, größter nicht-EN-Markt

**Workflow:**
1. Fiverr-Reviewer für jede Sprache (~€30 pro Seite pro Sprache)
2. Reviews als Diff auf GitHub, merge über Dictionary-Edit
3. Flag in `dictionaries/{locale}.json._meta.reviewed_by_native_speaker: true`
